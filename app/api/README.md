# app/api/

**Слой представления (Presentation Layer).** Отвечает исключительно за HTTP: приём запросов, валидацию входных данных через Pydantic-схемы, вызов сервисов и формирование ответов. Никакой бизнес-логики здесь нет.

## Структура

```
api/
├── v1/
│   ├── routers/        # Endpoint-роутеры, сгруппированные по модулям
│   └── dependencies/   # FastAPI Depends-провайдеры
└── middleware/         # Middleware-компоненты
```

## Что сюда попадает

- `APIRouter` с эндпоинтами (`GET /orders`, `POST /companies`, и т.д.)
- Преобразование Pydantic-схем запроса → вызов сервиса → Pydantic-схема ответа
- HTTP-статус коды и заголовки ответа
- FastAPI `Depends` для внедрения зависимостей (аутентификация, пагинация, текущий пользователь)
- Middleware (CORS, логирование, обработка ошибок)

## Чего сюда НЕ попадает

- Бизнес-логика и правила валидации домена
- Прямые обращения к БД или внешним сервисам
- ORM-модели или SQL-запросы

## Версионирование API

API версионируется через URL-префикс (`/api/v1/`). При появлении breaking changes создаётся директория `v2/` с новыми роутерами, `v1/` продолжает работать параллельно до завершения миграции клиентов.
